# SPACE SHOOTER

Unityで制作した、2D縦スクロール弾幕シューティングゲームです。
大量の敵と弾幕が飛び交う中での「パフォーマンスの最適化」と、チーム開発を見据えた「保守性の高いコード設計」を意識して開発しました。

## 💻 技術的なアピールポイント（推しコード）

本プロジェクトでは、特に以下のアーキテクチャ設計とパフォーマンス・チューニングに力を入れています。
各見出しのリンクから、実際のソースコードをご覧いただけます。

### 1. [Object Poolパターンによるメモリ管理とGC対策]
シューティングゲーム特有の「大量の弾の生成・破棄」によるガベージコレクション（GC）スパイクと処理落ちを防ぐため、`Instantiate` / `Destroy` を避け、`SetActive` を用いたオブジェクトプールを実装しました。
* **関連スクリプト:** `PlayerShooter.cs`, `Bullet.cs`

### 2. [列挙型(enum)を用いた、拡張性の高い状態管理]
アイテムの種類や敵のステータス管理において、マジックナンバー（0や1などの直接的な数値）を排除し、`enum` を採用しました。
これにより、Switch文を用いた可読性の高い条件分岐を実現し、プランナーがUnityのInspector上で直感的に設定できる「ミスが起きにくい（チーム開発に強い）設計」にしています。
* **関連スクリプト:** `Item.cs`

### 3. [単一責任の原則に基づくデータクラスの分離]
ゲーム全体の進行フラグやトータルスコアを管理する `GameData` と、リザルト画面での内訳表示用に各ステージのスコアのみを一時保持する `ScoreData` を、別々の static クラスとして分離しました。
「1つのクラスは何でもできる神クラスにしない」というオブジェクト指向の基本原則を徹底しています。
* **関連スクリプト:** `GameData.cs`, `ScoreData.cs`

### 4. [エラーハンドリングとカプセル化]
外部（Managerクラスなど）からパラメータを流し込む際、変数を直接 `public` で書き換えるのではなく、`Initialize` メソッド等を経由して安全に値を渡すカプセル化を行っています。
また、必須コンポーネントの取得漏れや背景画像の枚数不足など、エディタ上での設定ミスを検知して `Debug.LogError` で分かりやすく出力するチーム開発向けのエラーハンドリングを実装しています。
* **関連スクリプト:** `Stage2WaveManager.cs`, `InfiniteBackgroundScroller.cs`
